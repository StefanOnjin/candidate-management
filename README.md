## Candidate Management - Implementation Notes
One of the main challenges was keeping the data consistent and avoiding unintended duplicates. Candidates and skills have a many-to-many relationship, because one candidate can have multiple skills and the same skill can belong to multiple candidates. To solve this, I introduced the CandidateSkill join table instead of storing skills as plain text inside the candidate entity. Another related issue was case sensitivity, where values like English and english, or two versions of the same email address with different letter casing, could be treated as different records. To prevent this, I used PostgreSQL citext for case-insensitive uniqueness on candidate email addresses and skill names. I also added validation in the service layer so the API can return clear error messages before the database constraint is reached.

I chose a simple layered architecture with service and repository layers because it seemed like the best balance for this task. I considered using Clean Architecture, a modular monolith, or a simpler structure within a single Web API project. Clean Architecture and a modular monolith would provide better organization for a larger system, but for this small CRUD-focused task, they would probably be unnecessarily complex. On the other hand, I did not want to keep all logic directly inside the controllers. That is why I chose a layered approach, where controllers handle HTTP requests, services contain business logic and validation, and repositories communicate with the database. This kept the project simple, readable, and professional enough for future extension.


The backend should be started using the IIS Expres profile in Visual Studio, since the React client is configured to communicate with the API through: https://localhost:44366/api
